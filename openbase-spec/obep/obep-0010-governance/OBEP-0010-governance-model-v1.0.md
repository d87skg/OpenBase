# OBEP-0010: Governance Model v1.0

## 1. 前言

### 1.1 目的

本规范定义 OpenBase 标准化组织的治理模型，包括组织架构、决策流程、角色职责、投票机制和争议解决机制。治理模型是 OpenBase 生态系统长期健康发展的**制度保障**。

### 1.2 范围

本规范涵盖：

- 组织架构与层级
- 角色定义与职责
- 决策流程（RFC/OBEP 流程、共识流程）
- 投票与批准机制
- 争议解决机制
- 治理文档的版本管理
- 治理流程的透明度

本规范不涵盖：

- 具体协议定义（参见 OBEP-0001 ~ OBEP-0006）
- 兼容性政策（参见 OBEP-0007）
- 版本管理政策（参见 OBEP-0009）

### 1.3 符合性关键词

本规范使用 RFC 2119 关键词：

- **MUST**：必须遵守
- **MUST NOT**：禁止行为
- **SHOULD**：建议遵守
- **SHOULD NOT**：不建议行为
- **MAY**：可选行为

## 2. 组织架构

### 2.1 组织层级

OpenBase 标准化组织采用三层架构：

```text
┌─────────────────────────────────────────────────────────────────┐
│                    Steering Committee                          │
│              (战略方向 + 最终决策权)                            │
├─────────────────────────────────────────────────────────────────┤
│                    Technical Committee                         │
│              (技术评审 + 规范批准)                              │
├─────────────────────────────────────────────────────────────────┤
│                    Working Groups                              │
│              (具体规范制定 + 实现)                              │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 组织角色

| 角色 | 所属层级 | 职责 |
| :--- | :--- | :--- |
| **Steering Committee** | 最高层 | 战略方向、最终决策、争议仲裁 |
| **Technical Committee** | 中层 | 技术评审、规范批准、标准认证 |
| **Working Group Lead** | 执行层 | 领导具体工作组 |
| **Working Group Member** | 执行层 | 参与具体规范制定 |
| **Maintainer** | 执行层 | 维护具体实现 |
| **Contributor** | 执行层 | 贡献代码和文档 |
| **Community Member** | 社区 | 参与讨论与反馈 |

## 3. 角色定义与职责

### 3.1 Steering Committee（指导委员会）

**组成**：

- 由 5-9 名成员组成
- 成员来源：主要贡献者、组织代表、独立专家
- 任期：2 年，可连任（最多连任一次）

**职责**：

| 职责 | 描述 |
| :--- | :--- |
| 战略方向 | 制定 OpenBase 长期发展战略 |
| 最终决策 | 对重大争议做最终裁决 |
| 治理变更 | 批准治理模型的变更 |
| 生态系统 | 促进 OpenBase 生态发展 |
| 经费管理 | 管理社区基金（如有） |

**决策方式**：

- 共识优先（Consensus First）
- 无法达成共识时，多数投票（≥ 2/3 通过）

### 3.2 Technical Committee（技术委员会）

**组成**：

- 由 7-13 名成员组成
- 成员来源：各工作组长、核心维护者、领域专家
- 任期：1 年，可连任（最多连任两次）

**职责**：

| 职责 | 描述 |
| :--- | :--- |
| 技术评审 | 评审 OBEP 提案 |
| 规范批准 | 批准规范冻结 |
| 标准认证 | 管理认证计划 |
| 版本管理 | 决定版本发布 |
| 兼容性决策 | 裁决兼容性问题 |

**决策方式**：

- 技术共识（Lazy Consensus）
- 有争议时，简单多数投票（≥ 50% + 1）

### 3.3 Working Groups（工作组）

工作组是 OpenBase 标准制定的核心执行单元。每个工作组负责一个或多个相关规范领域。

**初始工作组**：

| 工作组 | 职责 | 对应规范 |
| :--- | :--- | :--- |
| Evidence WG | Evidence Schema 维护与演进 | OBEP-0001 |
| Replay WG | Replay Protocol 维护与演进 | OBEP-0002 |
| Trust WG | Trust Model 维护与演进 | OBEP-0003 |
| Certificate WG | Certificate Schema 维护与演进 | OBEP-0004 |
| Registry WG | Registry Protocol 维护与演进 | OBEP-0005 |
| Wire Protocol WG | Wire Protocol 维护与演进 | OBEP-0006 |
| Conformance WG | Conformance Test Suite 维护与演进 | OBEP-0008 |
| Interoperability WG | 跨运行时互操作性 | 待定 |
| Security WG | 安全与隐私 | 待定 |

**工作组职责**：

| 职责 | 描述 |
| :--- | :--- |
| 规范维护 | 维护和更新相关规范 |
| 接收提案 | 接收社区 OBEP 提案 |
| 技术讨论 | 讨论技术问题 |
| 决策建议 | 向 Technical Committee 提建议 |
| 实现验证 | 验证规范的可行性 |

### 3.4 Maintainer（维护者）

**职责**：

| 职责 | 描述 |
| :--- | :--- |
| 代码维护 | 维护参考实现（OpenClaw） |
| 文档维护 | 维护项目文档 |
| 社区支持 | 回答社区问题 |
| PR 评审 | 评审并合并 Pull Request |
| 发布管理 | 管理版本发布 |

### 3.5 Contributor（贡献者）

**职责**：

| 职责 | 描述 |
| :--- | :--- |
| 代码贡献 | 提交代码和文档 |
| 提案提交 | 提交 OBEP 提案 |
| Bug 报告 | 报告问题 |
| 社区参与 | 参与讨论和评审 |
| 测试 | 参与测试和验证 |

## 4. 决策流程

### 4.1 决策类型

| 决策类型 | 负责方 | 决策方式 |
| :--- | :--- | :--- |
| 规范变更 | 对应工作组 → Technical Committee | 共识 + 投票 |
| 治理变更 | Steering Committee | 共识 + 投票 |
| 版本发布 | Technical Committee | 简单多数 |
| 成员加入 | Steering Committee | 共识 |
| 争议裁决 | Steering Committee | 多数投票 |
| 日常维护 | Maintainer | Lazy Consensus |

### 4.2 共识流程

OpenBase 的默认决策方式是基于共识的决策。

**共识流程步骤**：

```text
1. 提案发起
   ↓
2. 公开讨论（最少 7 天）
   ↓
3. 合并共识（记录反对意见）
   ↓
4. 宣布决定
   ↓
5. 记录决策（ADR）
```

**共识决策规则**：

| 情况 | 处理方式 |
| :--- | :--- |
| 无反对 | 接受 |
| 有反对，但非实质性 | 接受，记录反对意见 |
| 有实质性反对 | 延长讨论，寻求妥协 |
| 无法达成共识 | 升级至上级委员会 |
| 紧急情况 | 快速决策，事后确认 |

### 4.3 投票流程

当共识无法达成时，进行投票。

**投票类型**：

| 类型 | 要求 | 适用场景 |
| :--- | :--- | :--- |
| 简单多数 | > 50% | 日常决策 |
| 超级多数 | ≥ 2/3 | 重大决策、规范冻结 |
| 一致同意 | 100% | 治理变更、战略决策 |

**投票流程**：

```text
1. 发起投票（公告 + 投票期）
   ↓
2. 投票期（5-10 天）
   ↓
3. 计票
   ↓
4. 公布结果
   ↓
5. 执行决定
   ↓
6. 记录决定
```

### 4.4 Lazy Consensus（懒惰共识）

适用于日常维护决策。

**规则**：

- 如果 72 小时内无人反对，则视为通过。
- 若有反对意见，进入讨论流程。
- 适用于：Bug 修复、小改进、文档更新。

## 5. OBEP 流程

### 5.1 OBEP 生命周期

```text
┌────────────┐
│   Draft    │   ← 提案发起
└──────┬─────┘
       ▼
┌────────────┐
│   Review   │   ← 工作组评审（≥ 7 天）
└──────┬─────┘
       ▼
┌────────────┐
│   Accept   │   ← Technical Committee 批准
└──────┬─────┘
       ▼
┌────────────┐
│  Implement │   ← 实现与测试
└──────┬─────┘
       ▼
┌────────────┐
│   Frozen   │   ← 规范冻结
└──────┬─────┘
       ▼
┌────────────┐
│  Superseded│   ← 被新 OBEP 取代
└────────────┘
```

### 5.2 OBEP 状态

| 状态 | 描述 |
| :--- | :--- |
| `Draft` | 初始状态，正在编写 |
| `Review` | 公开评审中 |
| `Accepted` | 已批准，等待实现 |
| `Implemented` | 已实现，等待冻结 |
| `Frozen` | 已冻结，不可变更 |
| `Superseded` | 被取代 |
| `Rejected` | 被拒绝 |
| `Withdrawn` | 作者撤回 |

### 5.3 OBEP 评审要求

| 要求 | 说明 |
| :--- | :--- |
| 评审期 | 最少 7 天 |
| 公开 | 所有评审公开进行 |
| 记录 | 所有评审意见必须记录 |
| 回应 | 提案作者必须回应所有意见 |

## 6. 会议机制

### 6.1 会议类型

| 会议类型 | 频率 | 参与方 |
| :--- | :--- | :--- |
| Steering Committee 会议 | 每月 | Steering Committee |
| Technical Committee 会议 | 每月 | Technical Committee |
| Working Group 会议 | 每两周 | 各工作组 |
| 社区全体会议 | 每季度 | 所有人 |

### 6.2 会议规则

| 规则 | 说明 |
| :--- | :--- |
| 公开 | 所有会议公开 |
| 记录 | 会议记录公开 |
| 议程 | 提前 48 小时公布 |
| 参与 | 任何人可旁听 |
| 决策 | 会议决策需在会后确认 |

### 6.3 异步决策

OpenBase **鼓励**异步决策（GitHub Issues、邮件列表），以减少会议依赖。

## 7. 争议解决

### 7.1 争议升级路径

```text
工作组内部讨论
      ↓
工作组 Lead 协调
      ↓
Technical Committee 介入
      ↓
Steering Committee 仲裁
      ↓
外部调解（如需）
```

### 7.2 争议解决原则

| 原则 | 描述 |
| :--- | :--- |
| 透明 | 所有争议公开讨论 |
| 中立 | 所有参与者平等对待 |
| 共识优先 | 优先寻求共识 |
| 记录 | 所有决议记录在案 |
| 可申诉 | 任何决议可申诉至上级 |

### 7.3 争议解决时间线

| 阶段 | 时间 | 说明 |
| :--- | :--- | :--- |
| 工作组讨论 | 2 周 | 工作组内解决 |
| TC 介入 | 1 周 | Technical Committee 裁决 |
| SC 仲裁 | 1 周 | Steering Committee 最终裁决 |

## 9. 成员管理

### 9.1 成员加入

**加入条件**：

| 条件 | 说明 |
| :--- | :--- |
| 认同使命 | 同意 OpenBase 使命 |
| 遵守准则 | 遵守行为准则 |
| 贡献 | 有实质性贡献 |
| 推荐 | 由现有成员推荐 |

**加入流程**：

```text
1. 申请（提交介绍）
   ↓
2. 评审（现有成员评审）
   ↓
3. 投票（≥ 2/3 通过）
   ↓
4. 接受（加入组织）
```

### 9.2 成员退出

| 方式 | 说明 |
| :--- | :--- |
| 自愿退出 | 个人主动退出 |
| 任期结束 | 角色任期结束 |
| 违反准则 | 违反行为准则 |
| 不活跃 | 长期不参与 |

### 9.3 成员权利与义务

**权利**：

| 权利 | 说明 |
| :--- | :--- |
| 投票权 | 参与决策投票 |
| 提案权 | 提交 OBEP 提案 |
| 选举权 | 选举委员会成员 |
| 被选举权 | 被选举为委员会成员 |

**义务**：

| 义务 | 说明 |
| :--- | :--- |
| 参与 | 积极参与组织活动 |
| 专业 | 保持专业态度 |
| 中立 | 以项目利益为重 |
| 保密 | 遵守保密规则 |

## 11. 示例

### 11.1 决策示例：新增规范

**场景**：社区提议新增 "Semantic Layer Specification"。

**流程**：

```text
1. 社区成员提交 OBEP-0011
   ↓
2. 工作组评审（7 天讨论）
   ↓
3. 技术委员会投票（5 天）
   ↓
4. 通过（8/10 赞成）
   ↓
5. 成立 Semantic Layer Working Group
   ↓
6. 实现并冻结
```

### 11.2 决策示例：争议

**场景**：两个工作组对 Evidence Schema 变更意见不一致。

**流程**：

```text
1. Evidence WG 提出变更提议
   ↓
2. Replay WG 提出反对（会影响重放）
   ↓
3. 两组合并讨论（2 周）
   ↓
4. 无法达成共识
   ↓
5. 升级至技术委员会
   ↓
6. 技术委员会裁决
   ↓
7. 决策记录在 ADR
```
